# PRD: 重新向量化端点（kb_reembed）

> **状态**：待实现（Phase 1.4）
> **模块**：docrank-memory, docrank-mcp
> **日期**：2025-03-08

---

## 1. 背景与目标

### 1.1 问题陈述

当用户升级 Embedding 模型（如从 BGE-M3 切换到更新的模型）时，历史数据的向量与新模型不兼容，导致检索质量下降。目前没有批量重建向量的途径，用户只能删除所有数据重新写入。

### 1.2 目标

提供 `/mcp/kb_reembed` 端点，对知识库中所有现有 chunk 用当前配置的 Embedding 模型重新生成向量并写回，无需重新解析文档。

### 1.3 非目标

- 不重新分块（chunk 文本不变，只重建向量）
- 不支持增量（每次全量重建）
- 不支持并发调用（写入期间锁定）

---

## 2. 用户故事

```
作为开发者，
我希望升级 Embedding 模型后能一键重建所有向量，
以便不丢失已解析的文档内容，同时用上新模型的向量质量。
```

---

## 3. 功能需求

- **P0** `POST /mcp/kb_reembed`：触发全量重新向量化
- **P0** 从向量后端拉取所有 chunk 的文本（chunk_text）
- **P0** 用当前 `EmbeddingProvider` 批量重新生成向量
- **P0** 将新向量批量写回向量后端（upsert，保留其余字段不变）
- **P0** 响应返回：处理的 chunk 总数、耗时
- **P1** 支持分批处理（batch），避免大知识库时 OOM

---

## 4. 非功能需求

| 维度 | 要求 |
|------|------|
| 耗时 | 可能很长（分钟级），响应为同步（当前阶段）|
| 安全 | 不删除旧向量，upsert 覆盖，中途失败可重跑 |

---

## 5. 验收标准

- [ ] AC1：调用 `kb_reembed` 后，`kb_search` 结果与新模型向量一致
- [ ] AC2：响应中 `chunk_count` 等于重建前 `kb_stats.vector_chunks`
- [ ] AC3：中途失败后重跑不产生重复数据（upsert 幂等）
